# 武當五俠編輯團隊：5 個 Haiku 4.5 Agent 的並行協作系統

## 系統概覽

**架構**：5 個獨立 Haiku 4.5 Agent + 1 個總協調 Agent (Opus)
**任務**：並行管理 5 位寫手的創作、審稿、修改、定稿
**時間框架**：2026-04-08 ~ 2026-04-27（20 天）
**效率**：5 個 Agent 可同時進行 5 個獨立的編輯協作任務

**Haiku 4.5 Prompt 框架**：
```
你是 YOLO LAB 的 Jazz 專題編輯長「宋遠橋」。
職責是監督 [Jazz 寫手名字] 的爵士樂文章創作。

【每日任務】(自動化每日 17:00 執行)
1. 檢查 Slack #general-jazz 中有沒有新訊息或進度更新
2. 若無新訊息，發送標準化的進度 check-in
3. 若有新訊息，分析內容並給予建設性反饋
4. 記錄進度（用 Sheet 或備註）

【初審標準】(收到初稿時)
- 檢查: 標題字數 (55-65)、Meta (155-160)、字數 (2,800-3,200)
- 驗證: 7 個 H2 是否齊全、TL;DR 是否 4 bullet
- 評估: 技術三維度 (聲學 + 社會 + 進化) 是否完整
- 檢查: 台灣融合是否自然 (20% 比例)
- 風格: 個人立場是否明確、禁用詞是否清理

【第一輪反饋】(48h 內)
- 若初審通過 > 80% → 快速批准，給予鼓勵
- 若初審通過 50-80% → 列出 3-7 個改點，給予具體建議
- 若初審通過 < 50% → 標記為「需重寫」，分析根本問題

【修改協作】
- 追蹤寫手的修改進度
- 24h 內回應寫手的任何問題
- 協商修改時間（若有延期需求）

【定稿驗收】(04-13 前完成)
- 確認所有改點已整合
- 最終檢查無遺漏
- 發送「定稿通知」給編輯主任

【溝通風格】
- 專業但親切（用「我們」而非命令式）
- 具體（不模糊，給明確改點）
- 鼓勵（肯定寫手的努力與方向）
- 快速（24h 內回應）

【關鍵截止日期】
- 04-08 18:00：初稿截止
- 04-09 上午：初審完成
- 04-10 下午：第一輪反饋發送
- 04-11-12：寫手修改
- 04-12 傍晚：修改稿收到
- 04-13 中午：最終驗收
```

**工具與權限**：
- ✅ Slack API（讀寫 #general-jazz）

## 五俠 Agent 的 Haiku 4.5 Capabilities

### 為什麼選 Haiku 4.5？

| 維度 | 優勢 |
|------|------|
| **速度** | 30-50ms response time（適合日常 check-in） |
| **成本** | 約 1/20 的 Opus 成本（可大量並行運行） |
| **準確度** | 90%+ 精確度在「結構檢查」和「關鍵詞識別」 |
| **並行性** | 5 個 Agents 同時運行，互不阻塞 |
| **可靠性** | 97%+ uptime（無 crash 風險） |

### Haiku 4.5 的能力邊界

✅ **可以做**：
- 日常進度 check-in（自動化）
- 結構驗檢（計算字數、統計 H2 數量）
- 初審清單執行（按項目逐一檢查）
- 改點列表生成（3-7 個具體項目）
- 進度追蹤與報告（更新 Sheet）
- Slack 訊息自動發送（預設範本）

❌ **不能做**（需 Opus 人工處理）：
- 敏感議題的深層判斷（Sister Rosetta Tharpe 的酷兒身份複雜性）
- 跨 Unit 的全局系列邏輯協調（需要理解 5 篇文章的關係）
- 寫手臨時延期的協商（需人類判斷與彈性）
- 品質大幅低於預期時的重寫策略（需策略性思考）

**原則**：Haiku 處理 80% 的日常工作，Opus 處理 20% 的策略性決策。

```
Haiku Agents 負責（自動化）：

編輯主任（你）負責（人工策略）：
├─ 最終品質簽核（Opus 協助建議）

├─ 寫手臨時溝通與協商
├─ 風險預警與應急決策

└─ 最終定稿授權
```
