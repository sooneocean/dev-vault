## 核心發現 3 項

### 1️⃣ 內嵌圖片部分失敗追蹤設計不清晰（風險等級：高）

**問題**：
- 計畫提及「一篇文章有 N 張圖片，若部分失敗…」但缺乏明確的狀態設計
- HTML 修改追蹤使用「索引」而非「URL」，可能導致重複修改或漏處理

**影響**：
- ❌ Resume 時無法準確識別已修改的圖片
- ❌ 可能重複修改相同圖片的 alt text
- ❌ HTML 結構損壞風險

**改進建議**：
- ✅ 改用 **URL 作為冪等性鍵** 而非位置索引
- ✅ State 中記錄「每張圖片的處理結果」（url, status, altText, error）
- ✅ 參考 `ALT_TEXT_DESIGN_IMPROVEMENTS.md` 的改進 #1

**影響範圍**: Unit 4（內嵌圖片更新）

---

### 2️⃣ 缺乏認證前置驗證（風險等級：高）

**問題**：
- Unit 1 掃描階段無 API 健康檢查
- 實際 `state_alttext_featured.json` 顯示 401 認證失敗，導致整批次失敗

**影響**：
- ❌ 發現問題時已浪費大量時間和 API 調用額度
- ❌ 無法前置診斷認證/權限問題

**改進建議**：
- ✅ Unit 1 開始前執行 **API 健康檢查**（認證、媒體讀寫、文章讀寫）
- ✅ 測試一張圖片的完整流程（Vision 存取 + media 更新 + post 更新）
- ✅ 參考 `ALT_TEXT_DESIGN_IMPROVEMENTS.md` 的改進 #3

**影響範圍**: Unit 1（掃描）→ 所有後續 Unit

---

### 3️⃣ 內嵌圖片 HTML 修改缺乏失敗恢復策略（風險等級：中-高）

**問題**：
- HTML 解析失敗的文章被標記為 failed，但無失敗計數或冷卻期
- Resume 時可能陷入「無限重試」同一失敗文章

**影響**：
- ❌ 資源浪費（重複嘗試無法修復的問題）
- ❌ 用戶體驗差（長時間 hang）

**改進建議**：
- ✅ 實施 **失敗計數 + 冷卻期** 機制（3 次失敗後 24 小時冷卻）
- ✅ State 記錄 `retryCount`, `lastFailTime`, `status` (retry_pending|abandoned)
- ✅ 參考 `ALT_TEXT_DESIGN_IMPROVEMENTS.md` 的改進 #4

**影響範圍**: Unit 4（內嵌圖片更新）

---

## 次要發現 4 項

### 4️⃣ ALT Text 長度截斷策略不完整

**問題**：計畫提及截斷但無智能斷詞邏輯，直接截斷可能破壞語義

**改進建議**：參考改進 #2（三層次截斷策略）

---

### 5️⃣ Skipped 狀態分類不明確

**問題**：State 中無法區分「合理 alt」vs「裝飾性圖片」，resume 時判斷困難

**改進建議**：參考改進 #5（結構化 skipped 條目）

---

### 6️⃣ Rollback 驗證邏輯缺失

**問題**：還原後無驗證和修復邏輯，失敗時無恢復手段

**改進建議**：參考改進 #6（驗證 + 修復邏輯）

---

### 7️⃣ 與 Phase 4 SEO 批次無互斥機制

**問題**：計畫未提及防止同時運行，可能導致 API rate limit 爭奪

**改進建議**：參考改進 #7（執行時互斥鎖）

## 附錄：核心設計決策確認

**Q: 為什麼選擇 Claude Haiku 而非 Sonnet？**
A: ALT text 是 80-125 字元的短文本生成，Haiku 足夠且成本最低（$0.001/張）。2,728 張圖片成本僅 $2-4，經濟性優於 Sonnet（$0.01/張，成本 $27+）

**Q: 為什麼混用 REST v1.1 和 wp/v2 API？**
A: v1.1 用於媒體對象（`alt` 欄位），wp/v2 用於文章內容讀寫（需 raw HTML 和 Gutenberg 保護）。這是既有模式（internal-linker-v2.js 驗證）

**Q: 如何避免 rate limit 導致失敗？**
A: 採用保守策略（batch 5、delay 2000ms）基於 Phase 9 的 100% 成功率驗證。若仍遇 429，自動指數退避重試

**Q: Rollback 能恢復所有數據嗎？**
A: 是的。備份包含 featured_media 的原始 alt + 內嵌圖片的原始 content.raw，支援完整還原。建議保留備份至少 7 天
